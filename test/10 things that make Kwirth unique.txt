## 10 Amazing Things That Make Kwirth Unique

Over the last few weeks, we’ve been discussing **#Kwirth**, explaining how its real-time live-streaming infrastructure provides DevOps, SREs, and Admins with high-precision data through a powerful management tool for Kubernetes clusters. 

Today, we begin a deep-dive series on the specific features that make **#Kwirth** a unique tool in the ecosystem.

---

### 1/10 Object Search
**Situation:** You need to find objects that depend on or use a specific resource, such as a ConfigMap.
**Solution:** Launch a Magnify channel (or open Kwirth Desktop), go to **'Cluster > Overview > Search'**, enter the ConfigMap name, and instantly see the results.
**Time to solve:** 3 seconds.
**NOTES:** You can search for any Kubernetes artifact, not just ConfigMaps or standard configuration items.

---

### 2/10 Secret Search
**Situation:** You’ve detected unauthorized access to your database (PostgreSQL, MongoDB, Oracle, SQL Server, etc.) originating from your Kubernetes cluster using the 'dbappuser' credentials.
**Solution:** Launch a Magnify channel, access **'Cluster > Overview > Search'**, enter the user ID you’re looking for, and view the results.
**Time to solve:** 3 seconds.
**NOTES:** If your secrets are stored in an external manager (Azure KV, AWS Secrets Manager, HashiCorp Vault...), finding this link manually can often take several hours.

[VIDEO]

---

### 3/10 Real-time Log Filtering
**Situation:** You are looking for a specific message within a Pod log stream.
**Solution:** Launch a log channel (or open a log window in Magnify), type the text you want to filter, and see the results in real-time.
**Time to solve:** 2 seconds.
**NOTES:** You can filter logging data from multiple sources simultaneously. When you apply a filter, it processes both historical (past) messages and incoming (future) messages.

[VIDEO]

---

### 4/10 Real-time Log Aggregation
**Situation:** You need to debug logs from multiple Pods across different namespaces that belong to the same application.
**Solution:** Select all the Pods, Controllers, or Namespaces you want to review and launch a log channel (or the equivalent in a Magnify log window).
**Time to solve:** 5 seconds.
**NOTES:** Use different views for aggregation. For example, use the **NAMESPACE** view to aggregate streams from all Pods in a namespace, or select specific Controllers and containers.

[VIDEO]

---

### 5/10 Metrics Aggregation
**Situation:** You need to monitor the resource consumption of a specific set of computing resources within your workload.
**Solution:** Just like log aggregation, you can configure metrics dashboards that merge and aggregate data from various origins.
**Time to solve:** 20 seconds.
**NOTES:** You can fully customize the chart type, data retention depth, and dashboard content.

[VIDEO]

---

### 6/10 Start Diagnostics
**Situation:** A Pod (or set of Pods) generates excessive logs immediately upon startup due to connectivity, credential, or database issues, making it impossible to catch the initial error messages.
**Solution:** Launch a log channel, select **'Start Diagnostics'**, configure the limits, and start the channel.
**Time to solve:** 10 seconds.
**NOTES:** The log stream will automatically pause once the message limits are reached for all Pods, allowing you to analyze the boot sequence.

[VIDEO]

---

### 7/10 Filesystem Access
**Situation:** You need to review or edit the content of a `.env` file inside a running Pod.
**Solution:** Launch a **'Fileman'** channel, wait for the file manager to load, and navigate just as you would in your local file explorer. Download, view, edit, or upload as many files as needed.
**Time to solve:** 15 seconds.
**NOTES:** You can copy/move files between containers (**#Kwirth** handles the underlying instructions). You can also download entire folder structures.

[VIDEO]

---

### 8/10 Cluster Navigation
**Situation:** You are troubleshooting a volume issue in a Pod and need to trace the underlying resource chain.
**Solution:** In Magnify or Kwirth Desktop, follow a simple visual sequence: **'Pod details > Storage > PVC > PV > Volume Attachment'**.
**Time to solve:** 5 seconds.
**NOTES:** Full navigation is supported for all Kubernetes artifacts, including CRDs and CRD instances.

[VIDEO]

---

### 9/10 Customization
**Situation:** As a DevOps/SRE, you frequently launch Pods for diagnostics, shell sessions, or troubleshooting.
**Solution:** Configure a **'Custom Action'** that waits for a Pod to be ready and automatically launches a shell session. This action is saved for permanent use.
**Time to solve:** 20 seconds.
**NOTES:** You can configure unlimited actions. Actions can wait for various states (like an active HTTP/HTTPS server) before triggering, not just a shell.

[VIDEO]

### 10/10 Architectural Flexibility
**Situation:** You have diverse needs, such as: (1) Integrating with **#Backstage**, (2) Running a local server without a cluster, (3) Using a native desktop experience, or (4) Optimizing for low latency.
**Solution:** 1. Deploy **Kwirth External** alongside your **#Backstage** instance.
2. Use the **Dockerized version** of Kwirth for local development.
3. Install **Kwirth Desktop** (available for Windows & Linux).
4. For in-cluster performance, deploy **Kwirth** directly via Helm or manifests.
**NOTES:** **#Kwirth** components are modular. You can connect Kwirth Desktop to a Kubernetes-based Kwirth instance, or Kwirth External to a Dockerized setup, depending on your requirements.
